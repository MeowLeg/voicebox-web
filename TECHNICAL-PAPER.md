# 基于 Vue 3 的媒体 TTS 语音合成系统设计与实现

## 摘要

本文介绍了一套面向新闻媒体场景的文本到语音（Text-to-Speech, TTS）合成系统的前端实现方案。该系统以 Vue 3 为核心框架，结合 TypeScript、Tailwind CSS 等现代前端技术，构建了覆盖新闻内容获取、多引擎 TTS 合成、音色克隆、音频后处理及生成历史管理的完整工作流。系统支持 Qwen、Chatterbox、LuxTTS 等十余种开源 TTS 模型，实现了客户端录音、WAV 格式编解码、多分段长文本合成等关键技术。本文详细阐述了系统架构设计、音频处理管线、内容集成机制及工程实践中的技术取舍。

**关键词：** 文本到语音；Vue 3；Web Audio API；音色克隆；多模型调度；WAV 编解码

---

## 一、引言

随着大语言模型与语音合成技术的快速发展，媒体行业对自动化语音内容生成的需求日益增长。传统新闻播报高度依赖人工配音，存在成本高、效率低、灵活性差等问题。本文所述的"舟传媒科技部 TTS"系统旨在为内部新闻生产流程提供一站式的语音合成工具，支持报纸新闻与电视新闻两大内容源，覆盖从内容选择、音色配置、参数调整到音频生成与管理的完整闭环。

系统前端采用单体单页应用架构，后端基于 Python FastAPI 构建。本文聚焦前端部分，重点讨论以下技术要点：

1. 基于 Vue 3 Composition API 的无状态管理库架构模式
2. 纯客户端 WAV 音频录制与多文件合并管线
3. 多 TTS 引擎的统一调度与音色管理模式
4. 双内容源的差异化集成方案

## 二、系统架构设计

### 2.1 技术选型

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| 框架 | Vue 3（Composition API） | 3.4.x | 组合式 API 便于逻辑复用 |
| 语言 | TypeScript | 5.3.x | 类型安全，减少运行时错误 |
| 样式 | Tailwind CSS | 3.4.x | 原子类快速构建响应式界面 |
| 构建 | Vue CLI（Webpack） | 5.0.x | 成熟稳定，生态完善 |
| 后端 | FastAPI（Python） | — | 高性能异步框架，自动生成文档 |

系统未引入 Pinia 或 Vuex 等外部状态管理库。全部状态通过 `ref()` 和 `computed()` 在单一 `Home.vue` 组件内管理。这一取舍基于系统的内部工具定位——单页面、少路由的场景下，引入状态管理库带来的复杂度收益不高，反而增加心智负担。

### 2.2 应用结构

前端代码库结构精简，共 12 个源文件，2 个 Vue 组件：

```
src/
├── api/index.ts          # API 客户端、类型定义、WAV 工具函数
├── views/Home.vue        # 全部 UI 与业务逻辑（1564 行）
├── router/index.ts       # 单路由配置
├── App.vue               # 根组件（仅渲染 router-view）
└── main.ts               # 应用启动入口
```

整个应用仅定义了一个路由 `/`，映射到 `Home.vue`。这种"单组件"架构虽不符合大型项目的最佳实践，但对内部工具而言，避免了跨组件通信、Prop 钻取等问题，降低了调试成本。

### 2.3 网络代理架构

开发环境通过 Vue CLI devServer 配置双层代理：

- `/voicebox-web/*` → `127.0.0.1:17493`（TTS 后端）
- `/voicebox-web/articles/*` → `127.0.0.1:4029`（新闻内容服务）

生产环境由 Nginx 统一反向代理，消除跨域问题。前端所有请求以 `/voicebox-web` 为基路径，实际目标由代理层转发。

## 三、音频处理管线

系统的音频处理涉及三个关键环节：客户端录音、WAV 文件构建、多分段音频合并。

### 3.1 客户端录音

录音功能基于 Web Audio API 实现，流程如下：

1. 调用 `navigator.mediaDevices.getUserMedia({ audio: true })` 获取麦克风 `MediaStream`
2. 创建采样率为 24,000 Hz 的 `AudioContext`
3. 通过 `createScriptProcessor(4096, 1, 1)` 捕获实时音频缓冲区
4. 在 `onaudioprocess` 回调中提取左声道的 `Float32Array` 数据并累积

需要说明的是，`ScriptProcessorNode` 已被 W3C 标记为弃用，推荐使用 `AudioWorklet` 替代。但在当前内部使用场景下，`ScriptProcessor` 的实现更为简洁，且兼容性覆盖所有目标浏览器。后续迭代可考虑迁移至 `AudioWorklet`。

### 3.2 WAV 文件构建

`createWavBlob()` 函数在客户端将原始浮点采样数据转换为标准 WAV 文件。核心步骤包括：

**自动归一化。** 遍历所有采样点找出峰值振幅，当峰值超过 0.95 时施加增益补偿（目标 0.9），防止量化溢出导致爆音。

**RIFF 头部构造。** 手动编写 WAV 二进制结构：

```
偏移    内容
0-3     "RIFF" 标识
4-7     文件总大小 - 8（小端 uint32）
8-11    "WAVE" 标识
12-15   "fmt " 子块标识
16-19   子块大小（16，标准 PCM）
20-21   音频格式（1 = PCM）
22-23   声道数
24-27   采样率
28-31   字节率（采样率 × 声道数 × 位深 / 8）
32-33   块对齐（声道数 × 位深 / 8）
34-35   位深（16 bit）
36-39   "data" 子块标识
40-43   数据块大小
44+     音频采样数据
```

**量化转换。** 将 `Float32` 范围 [-1.0, 1.0] 的采样值线性映射为 16 位有符号整数（小端序），负数使用 `sample * 0x8000`，正数使用 `sample * 0x7FFF`，以正确处理 -32768 的不对称边界。

### 3.3 多分段音频合并

长文本合成时，系统通过 `splitText()` 将输入文本按标点符号智能切分为不超过 500 字符的段落，各段落独立生成音频后由 `concatWavBlobs()` 合并：

1. 解析第一个 WAV 文件的格式参数（采样率、声道数、位深）
2. 遍历每个 WAV 的 RIFF 子块，定位 `data` 数据区
3. 累加所有数据区长度，合成新头部
4. 按序拼接原始数据字节

该函数支持非标准 WAV 文件（含额外元数据子块），通过逐块扫描而非硬编码偏移来定位音频数据。

## 四、TTS 引擎调度与音色管理

### 4.1 多引擎架构

系统通过 `getEngine()` 函数将模型名称映射到引擎标识，支持以下模型：

| 引擎 | 模型 | 音色模式 |
|------|------|----------|
| Qwen TTS | 1.7B / 0.6B | 可克隆 |
| Qwen Custom Voice | 1.7B / 0.6B | 可克隆 |
| LuxTTS | — | 可克隆 |
| Chatterbox | TTS / Turbo | 固定音色 |
| TADA | 1B / 3B-ML | 固定音色 |
| Kokoro | — | 固定音色 |

模型管理分为三个生命周期状态：可用（未下载）→ 已下载 → 已加载（可生成）。用户在模型管理弹窗中可触发下载与加载操作，加载后的模型在模型下拉框中标记为"已加载"。

### 4.2 双音色模式

系统支持两种互斥的音色选择机制：

**音色克隆模式。** 当所选模型的 `supports_clone` 为 `true` 时，用户需从已创建的 `VoiceProfile` 中选择目标音色。新建音色需上传一段参考音频（可录音或上传文件）及其对应文字，后端据此提取声纹特征。

**预设音色模式。** 当 `supports_clone` 为 `false` 时，系统根据引擎类型拉取预设音色列表，用户从中选择性别、语言匹配的预设声音。

### 4.3 音频效果链

系统为每个音色档案维护一条可配置的效果链，支持 9 种音频效果：合唱/镶边、混响、延迟、压缩器、增益、高通/低通滤波、音调调整、语速调节。效果链以 JSON 数组形式存储，每条效果包含类型、启用状态及参数字典。前端通过弹窗 UI 提供开关控制与滑块调参，保存后通过 `PUT /profiles/:id/effects` 接口同步到后端。

## 五、新闻内容集成

系统对接两类新闻内容源，采用差异化处理策略：

### 5.1 报纸新闻

通过分页接口按站点 ID、文档状态、日期范围拉取文章列表，返回结构化数据（标题、作者、发布日期等）。选中文章后调用详情接口获取正文内容，填充至文本编辑框供用户编辑或直接生成。列表去重基于 `METADATAID` 字段实现。

### 5.2 电视新闻

电视新闻接口结构更为复杂。系统先拉取串联单列表，再逐单获取详情，从中提取节目条目。内容正文采用中文方括号标注分段（如 `【正文】`、`【导语】`），系统通过正则表达式 `/(?=【[^】]+】)/` 按标签切分，将每条段落渲染为独立的可勾选文本框。用户可自由选择需要合成的段落组合，勾选内容实时拼接为生成文本。

## 六、生成管线与异常处理

### 6.1 异步生成流程

语音生成采用"提交 + 轮询"模式：

1. 前端构建 `GenerationRequest`，包含文本、语言、引擎、音色等参数
2. 调用 `POST /generate` 提交任务，返回任务 UUID
3. 以 1 秒间隔轮询 `GET /generate/:id/status`，直到状态变为 `completed` 或 `failed`
4. 生成完成后，通过 `/audio/:id` 获取音频文件

整个流程由 `AbortController` 保护。用户点击"停止"按钮时触发 `abort()`，同时中断生成请求和状态轮询。

### 6.2 SSE 响应解析

系统后端可能返回标准 JSON 或 Server-Sent Events 格式。`parseResponse()` 函数根据响应头 `Content-Type` 自动识别：

- `text/event-stream`：提取 `data:` 前缀后的 JSON 载荷
- `application/json`：直接调用 `res.json()`

### 6.3 超时与错误隔离

历史记录删除操作设置 10 秒超时，防止后端无响应导致请求挂起。后端健康检查每 30 秒执行一次，3 秒超时，通过页面右上角指示灯可视化后端连接状态。后端离线时，生成按钮自动禁用。

## 七、工程实践与展望

### 7.1 类型安全

全项目采用 TypeScript 严格模式，API 层定义了完整的接口类型（`VoiceProfile`、`GenerationRequest`、`ModelInfo` 等），编译时即可捕获参数类型不匹配的问题。Vue 组件内的 `ref` 均带有泛型参数，确保模板表达式类型推导正确。

### 7.2 改进方向

当前架构作为内部工具已经满足需求，但在以下方面仍有优化空间：

1. **音频录制迁移至 AudioWorklet。** `ScriptProcessorNode` 在主线程运行，长时间录音可能影响 UI 响应。AudioWorklet 在独立线程中处理音频数据，可避免此问题。
2. **组件拆分。** 当前 Home.vue 达 1564 行，建议将模型管理、音色创建、效果配置等弹窗拆分为独立组件，提升可维护性。
3. **引入状态管理。** 若后续扩展多页面或多模块，Pinia 可作为轻量级状态管理方案。
4. **生成进度 WebSocket。** 当前轮询模式存在延迟，可升级为 WebSocket 实时推送生成进度。

## 八、结语

本文介绍了一套面向媒体行业的 TTS 合成系统前端实现。系统通过合理的架构设计，在有限的代码量下实现了多引擎调度、音色克隆、客户端音频处理、双内容源集成等核心功能。纯客户端 WAV 编解码管线避免了额外的服务端依赖，多分段合并方案支持长文本合成。作为内部生产工具，该系统在保证功能完整性的同时，保持了较低的维护成本，体现了"适度架构"的工程理念。
